# 🛡️ YouTube AutoFlow V2: 프로젝트 히스토리 및 필수 제약 조건 (RULES.md)

이 문서는 과거부터 수많은 시행착오와 Git 커밋을 통해 쌓아 올린 **핵심 아키텍처와 절대 지워져서는 안 될 제약 조건들**을 기록한 "단일 진실 공급원(Single Source of Truth)"입니다. 앞으로 코드를 수정하거나 리팩토링할 때 이 규칙들이 훼손되지 않도록 반드시 참고해야 합니다.

---

## 1. AI 프롬프트 절대 제약 조건 (Global Negative Rules)
프롬프트를 전면 수정하더라도 아래의 룰은 `app.js` 최상단의 `GLOBAL_NEGATIVE_RULES` 변수로 보호되며 항상 포함되어야 합니다.

- **[채널 톤앤매너 & 설교 금지]** NO PREACHING: 대본의 방향성은 '팩트에 입각한 장기적인 안목'을 지향하되, 시청자를 훈계하거나 가르치려 들지 말 것. (예: "단단한 투자자가 되시길 바랍니다", "투자는 인내의 과정입니다" 등 명시적 훈계 및 감성적 클로징 절대 금지). 투자 철학은 객관적 분석 속에 자연스럽게 녹아들게 할 것.
- **[추상적 비유 금지]** NO ABSTRACT METAPHORS: 너무 추상적이거나 감성적인 비유, 뜬구름 잡는 문장을 쓰지 말 것. 인트로와 훅은 구체적이고 직관적이어야 함.
- **[반복 금지]** DO NOT REPEAT YOURSELF: 했던 말이나 통계, 훅(질문)을 앵무새처럼 무한 반복하지 말 것.
- **[타 채널명 언급 금지]** NO OTHER CHANNEL NAMES: 특정 채널명(예: 삼프로TV, 슈카월드 등), 타 유튜버 이름 절대 금지. (단, 요구된 자체 채널 인트로는 예외)
- **[논리적 전개]** PROGRESSIVE LOGIC: 서론-본론-결론이 흐름에 맞게 전진하며 멈춰있지 말 것.
- **[환각 금지]** NO HALLUCINATIONS: 반드시 제공된 원본 트랜스크립트 팩트 데이터에만 기반할 것.
- **[1줄 1문장 강제]** ONE SENTENCE PER LINE: 절대 문단(Paragraph)을 쓰지 말고 긴 문장을 짧고 임팩트 있는 한 줄짜리 문장들로 잘게 쪼갤 것.

## 2. 파트별 특수 지시사항 (Part-Specific Rules)
대본을 분할 생성(Chunking)할 때 훅과 인트로 파트에 들어가는 특수 규칙입니다.
- **[훅 (Hook)]**: 
  - 반드시 "여러분, ~ 나요?" 또는 "여러분, ~까?" 형태의 질문으로 시작할 것.
  - 질문은 절대 뜬구름 잡는 모호한(vague) 멘트가 아닌, 시청자의 뼈(정곡)를 찌르는 '매우 구체적이고 날카로운 팩트 기반의 페인포인트(Pain point)'를 짚어낼 것.
- **[인트로 (Intro)]**:
  - 초반부에 반드시 "결론부터 말하면~" 이라는 멘트를 넣어 시청 지속시간을 높일 것.
  - "이 영상을 보시면 ~ 를 알게 됩니다" 라는 멘트를 통해 시청자가 얻어갈 가치/정보를 명확히 제시할 것.
  - 마지막 문장은 반드시 다음의 채널 소개 멘트로 끝맺을 것: *"안녕하세요. 장기적인 안목으로 혼란스러운 시장 속에서 사실에만 집중하며 단단한 투자자가 되고 싶은 slowly but surely 입니다."*

## 3. 파이프라인 코어 히스토리 (Architecture History)
과거 버그들을 수정하며 완성된 핵심 파이프라인 로직들입니다. 시스템 고도화 시 절대 날아가면 안 됩니다.

1. **스마트 분할 생성 (Smart Chunked Generation) 및 Length Option 연동**
   - 대본을 생성할 때 토큰 초과 및 퀄리티 저하를 막기 위해 `훅 > 인트로 > 설명 > 결론` 구조를 쪼개어 순차적으로 생성함.
   - **(주의)** 훅 부분에서 질문이 30개씩 생성되는 무한 루프 버그를 해결하기 위해 파트별로 목표 줄 수를 다르게 할당하는 스마트 분량 조절 로직 적용 완 (훅: 3~5줄).
   - 본문(Body)의 경우 사용자가 선택한 **Length Option (`Short`, `Normal`, `Long`)**에 따라 목표 줄 수를 동적으로 조절함 (Short: 20~30줄, Normal: 40~60줄, Long: 80~100줄).
2. **API Key 자동 로테이션 (Quota Limit Bypass)**
   - `app.js`의 `callGemini` 함수 내부에는 Gemini API 키 여러 개를 쉼표(`,`)로 받아 429 에러(할당량 초과) 발생 시 자동으로 다음 키로 교체하는 핵심 로직이 탑재됨.
3. **1:1 매칭 구조 (1 Scene = 1 Sentence = 1 Prompt)**
   - 대본 창의 줄바꿈(`\n`)이 곧 1개의 독립된 장면(Scene)이 됨. 누락 방지 백업 로직(빈 곳 다시 찔러보기) 포함.
4. **시네마틱 vs 인포그래픽 자동 분류 (Data-driven Prompting)**
   - 대본 문장 내에 `%`, `억`, `달러`, `증가`, `수치` 등의 특정 데이터 키워드가 정규식으로 감지되면 무조건 네온 타이포그래피가 들어간 `[DATA/INFOGRAPHIC]` 스타일로 프롬프트를 빼고, 나머지는 `[CINEMATIC]` 스타일로 뺌.
5. **검토용 UI 2단계 정지 흐름 (Review & Confirm UI)**
   - 대본이 생성되자마자 이미지 프롬프트로 바로 넘어가던 과거의 폭주를 막기 위해, 대본 생성 후 텍스트 창에 멈춰서 검토/수정 및 텍스트 다운로드를 할 수 있도록 UI를 분리함.
6. **유튜브 백엔드 데이터 필터링**
   - 최신순 정렬(date), 1만 명 미만 구독자 필터, 영상 길이 세분화(1~5m, 5~20m) 기능.
7. **영상 주제(Video Topic) 주입 로직**
   - 사용자가 UI에 입력한 `영상 주제(videoTopicInput)` 텍스트를 대본 생성 프롬프트에 주입함.
   - 단순 언급을 넘어, **"모든 트랜스크립트 데이터를 해당 주제의 관점(Perspective)에서만 해석하고, 주제와 무관한 팩트는 과감히 버려라"**는 강력한 지시를 통해 대본의 뼈대를 잡도록 강제함.

## 🚨 AI 봇을 위한 엄격한 코드 수정 가이드 (CRITICAL AI INSTRUCTIONS)
이 프로젝트의 코드를 수정하는 AI 어시스턴트는 다음의 사항을 **절대적으로 준수**해야 합니다.
1. **[부분 수정 원칙]** 코드 수정 시, **기존의 잘 작동하는 부분(로직, 변수, 프롬프트 룰 등)은 절대 건드리지 마십시오.**
2. **[타겟팅 수정]** 오직 사용자가 명시적으로 요청한 부분만 `multi_replace_file_content` (Diff Edit) 도구를 사용하여 **정확하게 국소적으로만 수정**하십시오. 
3. **[전체 엎어치기 금지]** 프롬프트 문자열 덩어리나 함수 전체를 맹목적으로 덮어쓰기(Overwrite)하여 과거의 룰을 소실시키는 행위를 엄격히 금지합니다.
4. **[작업 전 필독]** 코드를 수정하기 전에 무조건 이 `RULES.md` 문서를 먼저 정독하여 기존 아키텍처와 충돌하지 않는지 확인한 후 진행하십시오.
